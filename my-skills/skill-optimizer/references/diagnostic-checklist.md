# Skill 诊断清单（problem-analysis rubric）

> skill-optimizer 的**问题分析阶段**用此清单。完整论据见同目录
> [skill-writing-handbook.md](skill-writing-handbook.md)；本文件是可操作的浓缩版。
> 每个维度给「症状 → 依据手册章节 → 修复方向」。

---

## 元规则（先于一切，决定哪条建议适用）

**不要把手册当教条照搬。同一条建议在 A 项目是优化，在 B 项目可能是倒退。**

优化任何 skill 前，先完成两件事：

1. **识别 skill 形态**（见下表）——形态决定哪些维度是关键、哪些不适用。
2. **探测宿主项目既有规范**——若项目根有自己的 skill/agent 指南、治理约束（如
   `CLAUDE.md`、`*-guide.md`、governance 目录、lint/audit 脚本），**先读它**。
   项目规则与手册冲突时，**以项目规则为准**，并在建议里显式标注这点。
   - 例：某项目用「机制硬门（audit/hook/schema）」管检查点，就**不要**反过来
     建议在 SKILL.md 里加散文检查点——那是退步。
   - 例：某项目禁止 LLM 输出加权小数总分，就**不要**建议加「8 维加权评分」。

### Skill 三形态（适用范围判定的核心）

| 形态 | 触发方式 | description 的真实作用 | 触发精准度关键? | 重点诊断维度 |
|---|---|---|---|---|
| **自动触发 skill** | description 语义匹配 | 决定何时被拉起 | **是，最关键** | 描述精准 + 反例触发 + 示例 |
| **编排型 agent** | 被编排器/上层显式加载为 system prompt | 给编排器/人看的文档 | 否（显式加载为主） | 输出契约 + 判断框架 + 长度 |
| **函数型桥接** | 被代码/agent 程序化调用 | I/O 契约 | 否（与触发无关） | 入参/出参 schema + 探活 + 安全 |

判定方法：看 description 里写没写"仅 X 调用 / 程序化调用"，看是否包脚本（`bridge.py`
等），看是否被列入某编排图。**判错形态 → 给错建议**（例如给函数型桥接优化"触发词"
毫无意义）。

## A. Description 与触发（自动触发 skill 最关键）

| 症状 | 依据 | 修复方向 |
|---|---|---|
| description 笼统（"处理代码迁移"） | §3.1 | 写清做什么 + 何时用 + 技术关键词 + 起点终点 |
| 内部黑话（TCC / QCMS 等缩写） | 反模式2 | 用通用语言描述功能，辅以技术关键词 |
| 无反例触发 | §10.2 | 补"不要在 X 时触发（用 Y skill）"——把"不做什么"写清往往比"做什么"更有效 |
| 多个 skill description 语义重叠 | §10.2 | 差异化关键词 / 加排斥说明 / 考虑合并 |

诊断动作：自造 ~20 条混合 query（一半该触发、一半不该），判断命中率（§3.1 触发评估）。

## B. 目标与适用判断（开头三件事）

- [ ] 开头说清 **做什么 / 为什么 / 何时不做**（§3.2）
- [ ] 给出"何时跳过"的前置检查（含具体命令，而非模糊描述）
- [ ] 起点和终点明确（从哪到哪）

## C. 指令语气与"为什么"

- [ ] 用祈使句直接下指令，不用"你应该/你可能需要"（§3.3）
- [ ] 关键约束**讲清原因**而非堆 MUST——AI 懂了道理才能应对没预想到的场景（§3.3）

## D. 示例 / Few-Shot（反模式3）

- [ ] 关键操作有 Before/After 或输入/输出示例（§3.4）
- [ ] 3-5 个示例，覆盖**典型 / 边界 / 错误**三类，最典型的放第一个（§3.5）
- [ ] 示例之间有差异，能展示不同处理分支
- 注：示例的价值是**让输出更稳定**。若宿主项目极度看重 token 预算，权衡时以
  "该示例是否服务输出稳定性"为取舍标准，而非一刀切删。

## E. 可视化（决策树 / 表格）

- [ ] 多分支逻辑用表格或 ASCII 决策树呈现，而非长段文字（§3.6）
- [ ] 能结构化的信息（配置字段、方案对比）尽量用表格（§5.1）

## F. 长度与模块化

| 症状 | 依据 | 修复方向 |
|---|---|---|
| SKILL.md > 500 行 | §4.1 | 拆主 skill + 子 skill / 子步骤文档 |
| 一个 skill 干 ≥3 件不相关的事 | 反模式1 | 单一职责，拆分 |
| 大段背景/历史沿革占正文 | 反模式6 | 背景外置 references/，正文只留"做什么/怎么做" |
| 复杂检查逻辑堆在正文 | §5.2 | 写成 scripts/，正文只调用 |

## G. 验证与检查点

- [ ] 有验证清单 + 可直接跑的检查命令（§9.1-9.2）
- [ ] 关键步骤之间有检查点，而非一口气做完才验证（反模式4 / §10.3）
- ⚠️ **但**：若宿主项目已用机制（CI/hook/schema/audit 脚本）管检查，**优先复用机制**，
  不要在 SKILL.md 里重复堆散文检查点（机制 > 文字）。

## H. 易错点与边界

- [ ] 显眼标出常见坑（误替换、类型不兼容、跨平台差异等）（§5.4）
- [ ] FAQ 处理"说不清"的边界情况（§5.5）
- [ ] 写死的数值改成判断规则 + 参考范围（反模式5）

## I. 脚本安全（§7，凡含可执行脚本必查）

- [ ] **无硬编码密钥**——用环境变量 / gitignore 配置文件（§7.1）
- [ ] 危险操作（删除/覆盖/DDL/mv）有确认或备份机制（§7.2-7.3）
- [ ] 外部输入做校验，防注入——**区分"指令"与"数据"，数据永不当指令执行**（§7.4）
- [ ] 文件路径不用未校验变量拼接（防穿越）；网络请求 HTTPS + 超时（§7.5）

## J. 外部服务集成（§6）

- [ ] 已有 MCP Server 的服务优先用 MCP；多 skill 复用 + 统一鉴权 → 封装 MCP；
      简单一次性调用 → 脚本里直接 HTTP（§6.2 决策树）
- [ ] 多个细碎、被广泛复用的"函数型"skill → 评估收敛为少数入口或 MCP，减少 Level-1 常驻

## K. 生命周期与评估（§8、§11）

- [ ] version 元数据 + 变更可追溯（§11.1）——按宿主项目约定，未必每个项目都需要
- [ ] 有评估用例（trigger-cases / quality-cases），改 description/步骤后重跑（§8.4.4）
- [ ] 拆出的子 skill 能独立使用（§4.4）

## 反模式速查（逐条对照）

| 反模式 | 一句话症状 | 解法 |
|---|---|---|
| 大杂烩 skill | 一个 skill 干三件事 | 拆分（F） |
| Description 黑话 | 全是内部术语 | 通用语言 + 技术关键词（A） |
| 没有示例 | 纯文字描述 | 加 Few-Shot（D） |
| 没有验证点 | 做完才检查 | 关键步骤加检查点（G，优先机制） |
| 写死数值 | 硬编码配置 | 给判断规则 + 参考范围（H） |
| 当 Wiki 写 | 背景 300 行正文 50 行 | 背景外置 references/（F） |

## 输出纪律（problem-analysis 阶段的产出）

- 按 **影响优先级排序**（先列致命/高频问题），每条给「症状 → 根因 → 依据章节」。
- 区分 **必修（影响正确性/安全）** vs **可选（打磨）**。
- 量化只用**确定性指标**（行数、缺字段数、密钥命中数）；**不要**给 skill 打加权小数总分。
- 凡建议与宿主项目既有规范冲突，**显式标注并以项目规范为准**。
